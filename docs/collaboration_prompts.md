- Текущий результат по `data/reports/quality_summary.json`:

  - юрлиц с observed lots: `13`;
  - core-лотов после дедупликации: `1533`;
  - источники core-лотов: `Сбербанк-АСТ` - `940`, `Росэлторг` - `593`;
  - лотов с раскрытой ценой: `382`;
  - товарных строк: `3308`, из них с unit price: `2310`;
  - ссылок на документы: `3982`;
  - документов с извлеченным текстом: `250`;

Главный риск: пытаться любой ценой увеличить число строк и потерять точность. В этом проекте высокий балл дает не только объем, но и доказуемость источников, воспроизводимость, качество фильтрации, честные ограничения и сильная аналитика.

```text

Перед действиями прочитай:
1. `README.md`

4. `docs/compliance_report.md`
5. `docs/etp_integration_audit.md`
6. ТЗ и заметки команды, если они доступны.

Работай как инженер-аналитик и исследователь открытых закупочных данных:
- сначала фиксируй текущий baseline и качество данных;
- затем выбирай самый выгодный следующий шаг по соотношению "баллы / риск / время";

Формат работы с Никитой:

Definition of done:

- данные в core очищены и дедуплицированы;
- источники описаны, включая rejected/probe-only;
- есть PostgreSQL DDL/views/marts;
- notebook содержит графики и выводы в формате Observation / Interpretation / Significance / Limitation;
- README кратко объясняет, что собрано, откуда, почему это надежно и где открытые данные закончились.
```

## Prompt 4. Работа с лимитами парсинга

```text
Если автоматический парсинг упирается в лимиты, антибот, нестабильный сайт или слишком большой объем, не продолжай одним огромным прогоном.

Project-specific rule: не запускать полный `python -m purchase_analysis.cli run-all` как способ парсинга новых данных без явного согласия Никиты. Для прироста данных использовать только source sprint / batch protocol: один источник, одно юрлицо или ИНН, ограниченный период или лимит страниц, raw evidence и accepted/rejected отчет.

Перейди на batch protocol:
- один источник;
- одно юрлицо или один ИНН;
- один период: месяц или квартал;
- лимит страниц за прогон;
- raw checkpoint после каждого батча;
- resume-файл с последней успешной страницей;
- отчет accepted/rejected после каждого батча.

Каждый батч должен быть идемпотентным: повторный запуск не создает дублей.

Если три батча подряд дают 0 accepted rows, остановись и сформулируй гипотезу:
- источник реально не содержит Sber-закупок за период;
- роль выбрана неверно;
- поиск ищет по оператору площадки, а не заказчику;
- нужен другой endpoint;
- данные доступны только после авторизации;
- сайт возвращает title mentions, которые нельзя безопасно добавить в core.
```

```text
Сфокусируй аналитический модуль на практичном сравнении цен типовых предметов.

Цель: показать не только динамику закупок, но и применение данных: где цена за единицу заметно выше внутреннего benchmark.

Алгоритм:
1. Используй `procurement_items.csv` и `mart_unit_price_benchmarks.csv`.
2. Выбери предметы с понятной единицей измерения и достаточным числом сравнений.
3. Сгруппируй похожие предметы нормализацией текста: бумага, ноутбуки, стулья, мониторы, картриджи, уборка/кв.м, ремонт/кв.м, если такие строки есть.

5. Отфильтруй только интерпретируемые выбросы, где единица измерения сопоставима.
6. Для каждого флага найди документы/ТЗ, если есть текст, и извлеки объясняющие признаки: характеристики, объем, бренд, доставка, регион, сроки.

Выход:
- таблица unit-price flags;
- 2-4 графика;
- комментарий Observation / Interpretation / Significance / Limitation;

```
