# Стандарт разработки парсеров источников V2 (ГОСТ)

Этот документ описывает единый архитектурный стандарт, которому должны соответствовать все скрипты-парсеры источников (ЕИС, B2B-Center, Lot-Online, TekTorg, ZakazRF, TenderPro). Старые скрипты переписываются под этот V2 стандарт.

## Обязательные пункты стандарта парсинга

### 1. Многоуровневый поиск (Формирование запросов)
Скрипт должен искать не только по ИНН и точному названию.
* Обязательная загрузка scope через `purchase_analysis.source_sprint.read_scope(...)` или напрямую через `entity_resolution.load_entity_scope(...)`, если это низкоуровневый тест.
* Обязательное использование `entity_resolution.build_search_terms(entity, source_system="generic")`.
* Эта функция автоматически генерирует список запросов, включающий ИНН, ОГРН, КПП, официальные и краткие названия, `brand_aliases`, `search_terms`, а также JSON `aliases` из `configs/entity_scope.csv`.
* Скрипт перебирает **каждый** из сгенерированных поисковых запросов.

### 2. Пагинация
Скрипт должен гарантированно переходить по всем страницам результатов поиска. Обработка должна прерываться, если на новой странице нет лотов.

### 3. Фильтрация и валидация (Классификация матчей)
Вся найденная выдача должна прогоняться через `entity_resolution.classify_entity_match(...)`.
* Каждый кандидат/лот получает статус `decision`: `accept` (точно наш), `review` (похоже, но нужно проверить), `reject` (чужой лот).
* Оставляем только те лоты/компании, где `decision.accepted` == `True`. Мусор отбрасываем. Отклоненные лоты не скачиваются.
* Совпадение по роли `operator/platform/supplier/seller/title_mention/text_mention` не принимается в core, даже если название похоже. Сбербанк-АСТ как площадка не является закупкой Сбера.

### 3.1. Выходной `items.csv`
Все source sprint скрипты должны писать `items.csv` через `source_sprint.write_items_csv(...)`.

Стандартные поля должны идти первыми:

```text
source_system, platform_section, entity_name, customer_query,
procedure_number, lot_number, subject, customer_name, customer_inn,
region, status, tender_type, price_rub, deadline_at, detail_url,
tags, published_at, application_deadline, method_name, currency,
organizer_name, organizer_inn
```

Дополнительные поля источника можно оставлять после стандартных.

### 3.2. Даты
Период проекта фиксирован: `2024-01-01` - `2025-12-31`.

Используй константы из `purchase_analysis.source_sprint`:

* `DATE_FROM_DMY`, `DATE_TO_DMY` для источников с `dd.mm.yyyy`;
* `DATE_FROM_DT`, `DATE_TO_DT` для локальных datetime фильтров;
* `DATE_SCOPE_START`, `DATE_SCOPE_END` для date-window logic.

### 4. Обогащение скоупа
Если по какому-то лоту мы уверены в совпадении, но у заказчика/организатора есть другие названия или доп. реквизиты, мы собираем их как кандидатов (`entity_resolution.propose_identity_enrichment` или вручную формируем строку).
* Скрипт не должен сам перезаписывать `entity_scope.csv`.
* Все найденные кандидаты записываются в отдельный файл `identity_enrichment_candidates.csv` в директорию спринта.
* `scripts/merge_aliases.py` не является обязательным шагом pipeline. Он dry-run по умолчанию и пишет JSON `aliases` только с `--apply`.

### 5. Захват файлов с лотов тендеров
Для площадок, где скачивание файлов не требует сложной капчи или авторизации (ЕИС, Lot-Online), скрипт скачивает их в папку `raw/documents/{lot_number}/`.
* При скачивании обязательно парсить заголовок `Content-Disposition` для правильного определения расширения файла, если сервер отдает бинарник без явного расширения в URL.
* Для площадок с жестким Cloudflare/Captcha (B2B Center) скачивание файлов пока игнорируется.

### 6. Пробный запуск перед полным парсингом
**ОБЯЗАТЕЛЬНО**: Прежде чем запускать парсер на полный скоуп юрлиц (full scope), необходимо проверить работоспособность скрипта на маленьком кусочке данных3. **Проба (Strict Probe):**
   - Перед полным парсингом обязателен запуск на ограниченном скоупе (1-2 энтити).
   - **ОБЯЗАТЕЛЬНОЕ УСЛОВИЕ:** Проба считается успешной **НЕ просто тогда**, когда скрипт не упал с ошибкой. Проба успешна только тогда, когда скрипт **нашел хотя бы для одного энтити нормальный результат (реальные лоты)**, успешно их классифицировал и скачал документы.
   - **МУЛЬТИ-ПОИСКОВАЯ ПРОБА:** Проверка работоспособности скрипта должна проводиться не только для поиска по ИНН. Парсер должен быть протестирован (выдать реальные лоты) для **КАЖДОГО** поддерживаемого типа поиска:
     * Поиск по ИНН
     * Поиск по ОГРН
     * Поиск по КПП (если поддерживается источником)
     * Поиск по текстовым Алиасам (Название, бренд)
   - Если источник API технически не поддерживает какой-то тип (например, ищет только по ИНН), это должно быть явно подтверждено, и проба проводится только по ИНН. В остальных случаях — тестируются все виды запросов до первого реального лота по каждому виду. Это доказывает, что логика парсинга HTML/XML работает корректно.
* Только после успешного позитивного тестового запуска и валидации результатов можно запускать полный парсинг источника.
